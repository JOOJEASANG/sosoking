# Security hardening

이 문서는 2026-07-17 보안 강화 변경의 운영 영향과 배포 후 확인 항목을 정리합니다.

## 적용된 변경

- 이메일/비밀번호 계정은 이메일 인증 완료 후 AI·프로필·소셜 기능 사용
- 문서 ID 경로 구분자 차단과 신규 사건 ID 난수화
- 사건 접수·사건명 추천·항소 실패 시 사용량 예약 환불 및 중단 예약 자동 복구
- 사건 이미지 실제 파일 시그니처 검사
- 판결 저장과 사건 완료 상태 원자적 커밋
- 새 사용자 판결은 무조건 비공개로 생성
- 공개/비공개 전환 서버 트랜잭션화 및 공개 전 판결·항소문 전체 개인정보 재검사
- 공개 판결문에서 소유자 UID와 비공개 첨부파일 경로 제거
- 공개 댓글에서 Firebase UID·가명 식별자 저장 제거
- 투표·댓글·신고·항소심 일일 제한과 쿨다운 적용
- 오늘의 AI 판결 날짜별 실행 잠금
- 프로필 사진 경로를 사용자별 `avatar.jpg`로 고정
- Firestore 핵심 데이터와 복구 예약 문서의 클라이언트 직접 쓰기 차단
- Hosting 기본 보안 헤더 적용
- 관리자 화면에 중단 복구, 공개 식별정보 정리, 구형 공개 ID 감사 도구 추가

## 배포 후 필수 확인

1. 이메일 인증 메일 발송 도메인과 템플릿을 Firebase Console에서 확인합니다.
2. Storage Rules는 별도 수동 워크플로로 배포합니다.
3. 관리자 페이지의 **AI 관리** 탭에서 **공개 식별정보 정리**를 한 번 실행합니다. 이 작업은 기존 공개 판결문과 댓글에 남은 UID·첨부 경로·작성자 식별 필드를 제거합니다. 이후에는 매일 오전 3시 40분에 자동 실행됩니다.
4. 같은 화면에서 **구형 공개 ID 감사**를 실행합니다. 확정 건수가 1건 이상이면 과거 공개 URL 자체에 Firebase UID가 포함된 사건이 있다는 뜻입니다. 자동 ID 변경은 기존 공유 링크와 투표·댓글 경로를 깨뜨릴 수 있으므로 이 변경에서는 진단만 수행합니다. 결과를 확인한 뒤 별도 이전 계획을 세웁니다.
5. 기존에 여러 이름으로 저장된 프로필 사진은 새 사진 저장 시 `avatar.jpg`로 교체됩니다. 과거 파일은 관리자가 별도로 정리할 수 있습니다.
6. 공개 전환이 개인정보 탐지로 차단될 때 실제 개인정보를 제거한 후 다시 시도합니다. 탐지는 안전을 우선하므로 일부 일반 문구가 차단될 수 있습니다.
7. Firebase App Check는 콘솔에서 웹 앱용 공급자와 사이트 키를 발급한 후 별도 적용해야 합니다. 저장소만으로 유효한 사이트 키를 생성할 수 없어 이번 변경에는 강제 적용하지 않았습니다.
8. Google Cloud Billing 예산 알림과 Gemini 사용량 경보를 별도로 설정합니다.

## 구형 사건 ID 처리 원칙

과거 사건 ID가 `{firebaseUid}_{timestamp}_{random}` 형식이면 URL만으로 내부 UID가 노출될 수 있습니다. 무조건 문서 ID를 바꾸면 기존 공개 링크, `result_reactions`, `court_comments`, 신고 기록과 관리자 참조가 끊길 수 있으므로 자동 이전하지 않습니다. 관리자 감사 결과를 바탕으로 다음을 포함한 별도 마이그레이션을 준비해야 합니다.

- 새 난수 ID로 `cases`와 `results` 복사
- 투표·댓글·신고·통계 하위 데이터 이전
- 구형 URL을 새 ID로 연결하는 별칭 문서와 리다이렉트
- 이전 완료 후 구형 문서 및 Storage 경로 제거

## 롤백 주의사항

Firestore Rules만 이전 버전으로 되돌리면 브라우저 직접 쓰기가 다시 열릴 수 있습니다. Functions, Firestore Rules, Hosting을 같은 커밋 기준으로 함께 배포해야 합니다. 공개 식별정보 정리를 실행한 후에는 삭제된 내부 식별 필드가 자동 복원되지 않습니다.
